807 Garden — Complete Technical Overview
Written for a Replit agent building a similar community companion app

What it is
A daily-habit companion mobile app (Expo/React Native) that wraps a nature metaphor around three personal intentions. Members name three things they want to tend every day, and the app tracks whether they did it. Points, streaks, a knowledge base, and a rewards economy sit on top of that core loop.

The Three Sisters (Corn, Bean, Squash) is 807's metaphor. Your community can swap this for anything triadic:

Your community	Metaphor	Three things
Makerspace	Three Tools	Build · Learn · Share
Fitness co-op	Three Pillars	Move · Rest · Fuel
Library	Three Practices	Read · Reflect · Connect
Mental health app	Three Anchors	Body · Mind · Heart
Faith community	Three Intentions	Prayer · Service · Stillness

The state engine — GardenContext.tsx
This is the heart of the app. Everything lives here. Key design decisions worth copying:

Local-first with server sync for bookmarks only. The daily habit state (streak, points, tending log, labels) is entirely in AsyncStorage — no server dependency for the core loop. Only bookmarked tips sync to the server. This means the app works offline and feels instant.

// Storage keys — increment version on breaking schema change
const STORAGE_KEY = "garden_state_v3";
const LEGACY_KEY  = "garden_state_v2";
// Per-user bookmark key — avoids cross-user pollution on shared devices
function tipsKey(userId: string) {
  return `garden_tips_v1_${userId}`;
}
State shape:

interface GardenState {
  tier: "community" | "sustaining" | "harvest" | "producer" | "sponsor";
  streak: number;
  points: number;
  lastTended: string | null;          // ISO date "2026-05-21"
  completedActivityIds: string[];     // reset daily via activityDate
  activityDate: string | null;
  sisterLabels: [string, string, string];  // user-named intentions
  sistersChecked: [boolean, boolean, boolean];
  sistersDate: string | null;
  tendedDate: string | null;
  tendedMethod: "watering_can" | "mindful" | null;
  tendingLog: TendingRecord[];        // last 30 days
  prunedToday: [boolean, boolean, boolean];
  pruningDate: string | null;
  sisterHacks: [string[], string[], string[]];  // personal notes per intention
  savedTips: SavedTip[];
}
Two tending modes — a pattern worth keeping:

"watering_can"  → one tap, all three checked, +10 pts  (quick, frictionless)
"mindful"       → tap each plant individually, +25 pts  (intentional, rewarded more)
Streak logic — one function, easy to adapt:

function computeStreak(prev: GardenState, today: string, alreadyTended: boolean): number {
  if (alreadyTended) return prev.streak;
  const yesterday = /* ISO date string for yesterday */;
  return prev.lastTended === yesterday ? prev.streak + 1 : 1;
}
Bookmark sync strategy (server-wins on sign-in, best-effort push on toggle):

Server has data → server replaces local (handles reinstalls, device switches)
Server empty → migrate from local per-user cache, push to server
Network failure → restore from per-user local key, retry next launch
Account switch → clear bookmarks immediately, re-sync for new user

Post-tending pruning flow — offered 1.2s after completion:

A PruningPromptSheet appears once per session
User can accept → PruningCarouselSheet (reflective prompts per intention)
Or dismiss → a nudge stays visible to re-open it
Pruning state (prunedToday, pruningDate) resets each day
Saved tips row — horizontal scroll of bookmarked tip chips. Long-press shows unsave/keep split confirmation (no modal needed).

Custom SVG component using react-native-svg. Three plant sprites (Corn, Bean, Squash) animate between states:

State	Visual

Earn screen — (tabs)/earn.tsx
Daily activity list. Activities reset each morning (checked via activityDate). Five built-in types:

type ActionKind = "webview" | "recipe_snap" | "market_confirm" | "knowledge_base";
const ACTIVITIES: Activity[] = [
  { id: "recipe_snap",    points: 15, actionKind: "recipe_snap"    },
  { id: "read_guide",     points: 10, actionKind: "webview",  webviewPath: "/guide"    },
  { id: "browse_knowledge", points: 10, actionKind: "knowledge_base" },
  { id: "take_learning",  points: 15, actionKind: "webview",  webviewPath: "/learning" },
  { id: "market_checkin", points: 25, actionKind: "market_confirm"  },
];
webview kind opens the web app in an in-app browser (expo-web-browser via openInBrowser()), then awards points when the user returns. No verification — honour system. Points awarded on return, not on actual completion.

market_confirm kind is a simple Alert confirm. Works for location-based actions with no GPS.

recipe_snap kind opens RecipeSnapSheet — camera/image-picker sheet. Currently stores locally; no backend endpoint to submit to.

Four sections:

Content files — the easiest thing to swap
lib/sisterCardPrompts.ts — 12 reflective prompts per intention (Corn/Bean/Squash). Replace with your community's prompts. getRandomTip(sisterIndex) avoids repeating the same prompt twice in a row.

lib/pruningTips.ts — 3 gardening tips per plant shown in the post-tending carousel. Completely metaphor-specific — swap for your domain's equivalent micro-advice.

API surface (server side)
Only three endpoints used by the mobile app:

GET  /api/tips              → paginated tip list (category, search params)
GET  /api/tips/saved        → { tipIds: number[] } for authenticated user
PUT  /api/tips/saved        → { tipIds: number[] } — full replace, not patch
GET  /api/tips/:id          → single tip detail
Auth: Clerk Bearer token in Authorization header. The API base is https://${EXPO_PUBLIC_DOMAIN}/api.

Improvements for 807 Garden — compounded back
These are gaps the codebase reveals, in priority order:

1. Tier sync from the API (medium effort, high value)
Tier is self-reported in profile. Every member already has a real tier in the 807 Benefits database via Square/Clerk. A GET /api/members/me call on sign-in should set tier authoritatively, not let users self-select. Prevents inflated access in the rewards menu.

2. Weather uses hardcoded Thunder Bay coordinates
expo-location is already installed but not used. One Location.getCurrentPositionAsync() call on first launch, stored in AsyncStorage, would make the weather genuinely local for members in Sioux Lookout, Dryden, or Deer Lake.

3. Streak validation lives only on device
A user who changes their phone's clock or reinstalls gets a corrupt streak. Since tendingLog already contains dated records, a simple server endpoint (POST /api/garden/sync-streak) that validates the log against the server's current date would make streaks trustworthy — useful before rewards become real.

4. Reward redemption needs a QR/code flow
Currently: Alert → "Show this to a staff member". Should generate a short-lived redemption code (server-issued, one-use, time-bounded) that displays as a large number or QR. The server-side model is trivial — a garden_redemptions table with code, userId, rewardId, redeemedAt.

5. Recipe snap has no backend
RecipeSnapSheet captures an image and title, completes the activity, but sends nothing to the server. A POST /api/garden/recipe-snaps endpoint accepting a base64 or multipart image would let you build a community recipe wall later. The camera flow is already built.

6. Pruning reflections aren't persisted
sisterHacks: [string[], string[], string[]] exists in state and has an addSisterHack action, but nothing in the pruning carousel calls it. After a pruning session, saving the user's typed response as a hack would give them a personal journal per intention — differentiated enough to be worth surfacing.

7. Push notification for daily streak
expo-notifications (not yet installed) + a POST /api/garden/register-push-token endpoint. One daily reminder at a user-configured time ("Tend your garden 🌱") would materially improve retention. The streak logic to calculate "hasn't tended today" is already available server-side if you sync the log.

8. Knowledge base offline cache
Tips are fetched fresh via TanStack Query on every visit. A simple staleTime: Infinity + AsyncStorage-backed persistence layer (or just bumping gcTime) means the knowledge base works on the train. The bookmark sync is already offline-resilient — tips should be too.
