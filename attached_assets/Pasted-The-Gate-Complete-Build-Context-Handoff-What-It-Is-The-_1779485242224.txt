What It Is
The Gate is a translation/compliance layer for a fictional organisation called Bright Side. Bright Side uses warm, intentional internal language ("neighbour", "guide", "host"). When communications leave â€” a letter, a regulatory notice, a banking statement â€” they must use legally and institutionally required terms ("resident", "agent", "operator"). The Gate sits at that threshold: it takes text written in Bright Side's voice, silently substitutes every required term, and returns the compliant version without asking the author to change how they speak.

The conceptual frame comes from the Codetry / Headwaters handbook (https://northern-store-plan.replit.app/codetry-handbook/). The Gate is one of the five Pioneer Path stations â€” the Both-Sides primitive. Two contexts (Bright Side = inside, Massity = outside), one plumbing. The tool is itself a working instance of the thing it describes.

Contract-First API Design
The OpenAPI spec at lib/api-spec/openapi.yaml is the single source of truth. From it, Orval generates:

lib/api-zod/src/generated/ â€” Zod request/response validators (used by api-server for input validation and output parsing)
lib/api-client-react/src/generated/ â€” TanStack Query hooks (used by the frontend)
To regenerate after spec changes:

pnpm --filter @workspace/api-spec run codegen

Critical Orval naming gotcha: Orval ignores $ref schema names and auto-names request body types after the operation. The spec uses CreateMappingInput, UpdateMappingInput, TranslateInput â€” but the generated Zod values (importable constants) are named CreateMappingBody, UpdateMappingBody, TranslateTextBody. Import these names in route handlers, not the spec names.

Boolean query params gotcha: zod.coerce.boolean() (which Orval generates for boolean query params) coerces the string "false" to true because it's a truthy string. The activeOnly param in /api/mappings is parsed manually: req.query.activeOnly === 'true'. Any future boolean query params must do the same.

Database Schema
mappings table
{
  id: uuid (PK, random)
  friendly_term: text NOT NULL          // the inside/Bright Side term
  required_term: text NOT NULL          // the outside/Massity term
  category: text NOT NULL               // one of 5 values (see below)
  rationale: text NOT NULL DEFAULT ''   // why this substitution is required
  case_sensitive: boolean DEFAULT false
  whole_word_only: boolean DEFAULT true
  is_active: boolean DEFAULT true
  created_at: timestamptz
  updated_at: timestamptz
}

Categories (fixed enum, not user-extensible): pragmatism | politics | regulations | privacy | banking

translations table
{
  id: uuid (PK, random)
  input_text: text NOT NULL
  output_text: text NOT NULL
  substitutions_count: integer DEFAULT 0
  substitutions: jsonb NOT NULL DEFAULT []   // SubstitutionRecord[]
  created_at: timestamptz
}

substitutions is denormalized JSONB. Each element is:

{
  mappingId: string   // UUID of the mapping row used
  from: string        // friendly term that was matched
  to: string          // required term that replaced it
  category: string    // category of the mapping
  rationale: string   // why â€” copied from mapping at time of translation
  count: number       // how many times this mapping matched in this translation
}

Rationale and category are intentionally snapshotted into the translation record â€” so history is stable even if a mapping is later edited or deleted.

The Substitution Engine
File: artifacts/api-server/src/lib/gate.ts

This is the load-bearing piece. It went through one significant correctness fix â€” the original design applied mappings sequentially against the mutating output (cascading), which could cause a substitution introduced by mapping A to be re-matched by mapping B. The final implementation is non-cascading, single-pass.

Algorithm (three phases)
Phase 1 â€” collect all matches against the ORIGINAL input.
Every active mapping runs its regex against the untouched input string. All match positions (start, end, which mapping, matched text) are collected into a flat list. Crucially, the output is never read during this phase â€” only the original input.

Phase 2 â€” resolve overlaps.
Matches are sorted by start position ascending, then by mapping precedence (longer friendly terms sort earlier because the candidates array is sorted by friendlyTerm.length descending). A simple cursor walk keeps the first non-overlapping match at each position and discards any that start inside a previously accepted match. Longest term wins on overlap.

Phase 3 â€” build output in one pass.
Walk the accepted match list, emit unchanged input segments between matches, emit replacements at match positions, tally counts per mapping. One string concatenation at the end.

Per-mapping features
caseSensitive: false (default): Case-insensitive regex match. At replacement time, preserveCase() is called: ALL CAPS â†’ ALL CAPS, Title Case â†’ Title Case, anything else â†’ replacement as-written.
wholeWordOnly: true (default): Pattern uses Unicode-aware negative lookbehind/lookahead: (?<![\p{L}\p{N}_])term(?![\p{L}\p{N}_]). This handles non-ASCII word boundaries correctly.
caseSensitive: true: Exact match, no case manipulation.
All friendly terms are regex-escaped before pattern construction â€” user-created terms with regex-special characters (+, (, ., etc.) will not crash or give wrong results.
What the engine does NOT do (current gaps)
No position tracking returned to the client â€” the frontend highlights by re-running a split-regex on the output, which is a lossy approximation. Works for all current seeded terms but would mis-highlight if a required term appears naturally in the original text.
No multi-lingual support beyond Unicode word boundaries (the regex engine is Node.js's V8 ICU-backed one).
No phrase-level context awareness (cannot say "replace 'fee' only when preceded by 'late'").

Key Engineering Decisions & Why
1. Contract-first (OpenAPI â†’ Orval) rather than code-first
The spec defines the contract between frontend and backend. Changes to the API require updating the spec and regenerating. This prevents drift between what the server serves and what the client expects. Enforced by TypeScript end-to-end â€” the generated Zod schemas validate both incoming requests and outgoing responses.

2. Non-cascading substitution engine
The naive approach (loop over mappings, apply each regex to the output in sequence) causes double-translation: a term introduced by mapping A can be caught by mapping B. e.g., if resident were also a friendly term, neighbourâ†’residentâ†’something_else. The three-phase approach (collect all matches against original, resolve overlaps, emit once) eliminates this class of bug entirely.

3. Longest-match-first overlap resolution
When two mappings could both match overlapping spans, the longer friendly term wins. This is the linguistically correct choice: binding agreement should beat agreement if both are in the glossary. Implemented by sorting candidates by friendlyTerm.length descending before building the match list â€” so longer-term matches get lower mappingIndex values and win the sort when starts tie.

4. Denormalized substitutions in JSONB
Each translation stores a snapshot of the substitutions made, including the rationale text, at the time of translation. This makes history stable: editing or deleting a mapping never corrupts past translations. The downside is you can't retroactively see which mapping a historical substitution came from if the mapping was deleted (the mappingId will be a dead reference). Accepted trade-off.

5. activeOnly parsed manually in the route
zod.coerce.boolean() (what Orval generates for boolean query params) treats the string "false" as true because it's truthy. The activeOnly param bypasses the Zod output and is parsed as req.query.activeOnly === 'true'. This is explicitly documented in replit.md. Any new boolean query params must use the same pattern.

6. No auth
The app is unprotected. Any user with the URL can read, create, edit, and delete mappings and history. This was an intentional scope decision â€” the target use case is a single organisation's internal tool. Auth is a clear next step (see gaps).

Gaps & Known Limitations
Correctness
Highlighting is approximate. The frontend highlights substituted terms by splitting the output string on the term. If the same word appears naturally in the output text (not as a substitution), it will be highlighted incorrectly. The correct fix is for the backend to return character-offset spans alongside the substitutions, which the frontend renders directly. This is a medium-effort backend + frontend change.

preserveCase is limited to UPPER / Title / lower. Mixed-case patterns like camelCase or mIddle get the lowercase branch (replacement as-written). Acceptable for natural-language text; would be wrong for code identifiers.

No plural/morphological awareness. neighbour and neighbours must be separate mappings. The seeded data handles this for the common case, but any future term needs its variants explicitly added.

No phrase-context matching. Cannot express "replace 'fee' only when it follows 'late'." Every match is context-free.

Features Not Built
No authentication or multi-tenancy. The glossary and history are shared by everyone who hits the URL. Adding Replit Auth or Clerk Auth is straightforward â€” the api-server is Express 5 with session support (SESSION_SECRET env var is already provisioned).

No soft-delete / deactivation on translations. Translations can only be hard-deleted. There is no archive or undo.

No bulk import of mappings. Adding 50 new terms requires 50 form submissions. A CSV/JSON upload route would be high-value.

No audit log. No record of who changed what mapping or when (only updated_at timestamp on the mapping row).

No category management. Categories are a hardcoded enum (pragmatism | politics | regulations | privacy | banking). Adding a new category requires a schema change, a codegen run, and a redeployment. User-defined categories would require a separate categories table and changes throughout.

No search on the History page. The list is just most-recent-first with a fixed limit of 50. No full-text search, no date filter, no filter by category.

No copy-to-clipboard on the Translate page. The translated output has no affordance for copying it. Users must select-all manually.

No diff view. The translate output shows substituted terms highlighted, but there is no side-by-side diff mode showing exactly what changed and where (beyond what the popovers show).

Dashboard top-substitutions query is N+1. GET /api/dashboard/top-substitutions unnests substitution JSONB across all translations, groups by mappingId, then issues a separate DB query per result row to look up the mapping. For a small number of results (default 8) this is acceptable. At scale, should be replaced with a join or a materialized view.

No pagination anywhere. History uses a limit param, mappings returns all rows. For production volumes, proper cursor-based pagination is needed.

No dark mode. The palette and Tailwind config support CSS variables, but no dark mode toggle is implemented.

No export from the Gate. There is no way to export the full glossary (as CSV, JSON, or PDF) for compliance documentation or audit purposes.

Infrastructure
No background mapping refresh on translate. POST /api/translate loads all mappings from the DB on every request with db.select().from(mappingsTable). For a small glossary this is fine. At scale (hundreds of mappings, high request volume), a cached/memoized mapping load would be needed.

Dev restart required for code changes. The api-server has no hot-reload. After any code change, the workflow must be restarted. Vite handles this for the frontend automatically.
